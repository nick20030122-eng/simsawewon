# 심사위원 챗봇 — 코드 기획서

## 1. 목적

바이브 코딩(Vibe Coding) 검증 도구입니다. 전통적인 코드 리뷰(문법·오타 지적)가 아니라, **인간이 작성한 기획 의도가 AI가 생성한 코드에 왜곡 없이 반영되었는지**를 판단합니다.

## 2. 입력 (2가지만)

| # | 입력 | 설명 |
|---|------|------|
| 1 | 코드 기획서 | 요구사항 및 성공 기준 (텍스트 또는 `.md`/`.txt` 파일) |
| 2 | 작성된 코드 | `.py` 파일 1개 이상 (복수 파일은 병합하여 전달) |

추가 컨텍스트(Git 이력, 대화 로그, 테스트 결과 등)는 **입력하지 않습니다**.

## 3. 출력

### 3.1 종합 점수 표

| 항목 | 점수 범위 |
|------|-----------|
| 의도 구현도 (Intent Alignment) | 0–100 |
| AI 호환성 (AI-Friendliness) | 0–100 |
| 예외 처리 & 안정성 (Robustness) | 0–100 |
| 과잉 구현 방지 (Clean State) | 0–100 |
| **종합** | 4항목 산술 평균 |

### 3.2 평가 후기 (3블록만)

1. **잘한 점** — bullet list
2. **리스크** — bullet list
3. **최종 한마디** — 1~2문장

### 3.3 Pass / Fail

- 종합 점수 ≥ Pass 임계값(기본 70) → **Pass**
- 미만 → **Fail**

### 3.4 출력 금지

다음은 **절대 출력하지 않습니다**:

- 디버깅 가이드
- 수정 코드 / 리팩토링 코드
- "이렇게 고치세요" 형태의 기술적 처방

## 4. 4대 채점 기준

### 4.1 의도 구현도 (Intent Alignment)

기획서의 핵심 요구사항·성공 기준이 코드에 빠짐없이, 왜곡 없이 구현되었는가?

### 4.2 AI 호환성 (AI-Friendliness)

함수·모듈 분리, 명확한 네이밍, 단일 책임 등 Cursor 같은 AI가 읽고 확장하기 좋은 구조인가?

### 4.3 예외 처리 & 안정성 (Robustness)

빈 입력, API 오류, 파일 파싱 실패 등 방어적 프로그래밍이 되어 있는가?

### 4.4 과잉 구현 방지 (Clean State)

기획서에 없는 기능·라이브러리·복잡한 추상화를 AI가 임의로 추가하지 않았는가?

## 5. UI 요구사항

| 영역 | 요구 |
|------|------|
| Sidebar | OpenAI 모델 선택, Pass 임계값 슬라이더 |
| Tab 1 — 채점 | 기획서 입력(텍스트/업로드), 코드 입력(텍스트/업로드), 「심사 시작」 버튼 |
| Tab 2 — 채점 기준 | 4대 기준 설명 (정적 Markdown) |
| 결과 표시 | Metrics(항목별·종합 점수), 점수표, 평가 후기, Pass/Fail 배지 |

## 6. 기술 요구사항

| 항목 | 내용 |
|------|------|
| 프레임워크 | Streamlit |
| LLM | OpenAI API (기본 모델 `gpt-4o`, Structured Outputs) |
| API 키 | `.env`의 `OPENAI_API_KEY` (UI 노출 금지) |
| 프로젝트 구조 | `app.py`(UI) + `judge/`(평가 로직) + `prompts/`(시스템 프롬프트) |

## 7. 예외 처리 (성공 기준)

| 상황 | 기대 동작 |
|------|-----------|
| API 키 없음 | 앱 중단 없이 에러 메시지 표시 |
| 기획서 또는 코드 비어 있음 | 심사 시작 차단 + 안내 메시지 |
| OpenAI API 오류 | 앱 중단 없이 에러 메시지 표시 |
| JSON 파싱/스키마 검증 실패 | 앱 중단 없이 에러 메시지 표시 |

## 8. 범위 외 (1차 미구현)

- 다중 사용자 인증
- Git hook / CI 자동 채점
- 채점 이력 DB 저장
