# 채점 제출 파일 양식 가이드

> **참가팀**이 채점 페이지에 올릴 **기획서 · README · 실행 코드**를 어떻게 준비하면 좋은지 정리했습니다.  
> 「무엇을 올리나」뿐 아니라 **왜 이 규칙이 있는지**, **어긋나면 어떤 일이 생기는지**도 함께 적었습니다.

---

## 1. 공통 규칙

| 규칙 | 설명 | 왜 필요한가요? |
|------|------|----------------|
| **파일 3개** | 기획서 + README + 실행 코드 각 1개 | AI가 **교차 검증**하기 때문 |
| **같은 프로젝트** | 세 파일이 같은 과제를 설명 | 다르면 「의도 구현도」·「README 정합성」 평가 불가 |
| **텍스트 형식** | `.md`/`.txt`(문서), `.py`(코드) | 채점 시스템이 **글자만** 읽도록 설계됨 |
| **UTF-8** | 한글이 깨지지 않는 저장 | 깨진 글은 AI가 내용을 잘못 이해함 |
| **실제 내용** | placeholder·빈 파일 금지 | 너무 짧으면 해당 분야 **심사 불가(0점)** |

### 「같은 프로젝트」가 어긋나면 어떻게 되나요?

**나쁜 예**

| 파일 | 내용 |
|------|------|
| 기획서 | A팀 CSV 대시보드 |
| README | B팀 할 일 관리 앱 설명 |
| app.py | C팀 연습용 코드 |

→ AI는 「기획과 코드가 맞는지」를 볼 수 없어 **의도 구현도·README 품질이 크게 하락**하거나 심사가 제한됩니다.

**좋은 예**

| 파일 | 내용 |
|------|------|
| 기획서 | CSV 대시보드 기능·성공 기준 |
| README | CSV 대시보드 실행 방법 |
| app.py | CSV 대시보드 Streamlit 코드 |

→ 세 파일이 **한 제품**을 가리키므로 의미 있는 점수가 나옵니다.

## 2. 기획서

### 어디에 / 무엇을

- **채점 페이지 → 「기획서」** 칸
- 확장자: `.md`(권장) 또는 `.txt`

### 왜 필요한가요?

기획서는 **「무엇을 만들기로 했는지」** 의 기준입니다.  
AI는 이 문서를 보고 **공공기관 적합성**(문제·해결·현장성)을 평가하고, 나중에 **코드와 맞는지**도 비교합니다.

### 반드시 들어가면 좋은 내용

| 항목 | 이유 |
|------|------|
| **목적·페인포인트** | 공공 현장 **어떤 문제**를 푸는지 |
| **핵심 기능 + 성공 기준** | 코드 평가 시 **체크리스트**가 됨 |
| **UI 요구** | 화면 구성이 기획과 맞는지 판단 |
| **예외 처리** | 빈 입력·오류 시 동작이 기획과 맞는지 |
| **범위 외** | 없는 기능을 코드에 넣었는지(과잉 구현) 판단 |

### 작성 예시 (목차)

```markdown
# [프로젝트 이름] — 기획서

## 1. 목적
- 어떤 문제를 해결하는 앱인지 2~3문장

## 2. 핵심 기능
| 번호 | 기능 | 성공 기준 |
|------|------|-----------|
| 1 | CSV 업로드 | 파일 1개를 올릴 수 있다 |
| 2 | 요약 표시 | 숫자 컬럼의 평균·합계를 보여 준다 |

## 3. 화면 구성
- 페이지 제목, Sidebar 여부 등

## 4. 예외 처리
| 상황 | 앱이 해야 할 일 |
|------|----------------|
| 파일 없음 | 안내 메시지 표시 |

## 5. 범위 외
- 이번에 만들지 않는 기능

## 6. 사용 기술
- Streamlit, app.py 한 파일 등
```

### 흔한 실수와 결과

| 실수 | 결과 |
|------|------|
| 3~4줄만 적음 | 공공기관 적합성 **심사 불가** 또는 매우 낮은 점수 |
| 성공 기준 없음 | 의도 구현도 평가가 **모호**해짐 |
| 다른 팀/다른 과제 내용 | 의도 구현도·정합성 **급격히 하락** |

## 3. README

### 어디에 / 무엇을

- **채점 페이지 → 「README」** 칸
- 보통 프로젝트의 `README.md`를 그대로 업로드

### 왜 필요한가요?

README는 **「제3자가 혼자 설치·실행할 수 있는지」** 를 봅니다.  
채점 시스템은 앱을 **직접 실행하지 않으므로**, README가 사실상 **「실행 검증 대용 문서」** 역할을 합니다.

### 반드시 들어가면 좋은 내용

| 항목 | 이유 |
|------|------|
| **설치 명령** (`pip install …`) | 환경 구성 가능 여부 |
| **실행 명령** (`streamlit run app.py`) | 진입점이 명확한지 |
| **접속 URL/포트** | 실행 후 어디로 들어가는지 |
| **기능 목록** | 기획서·코드와 **같은지** 대조 |
| **프로젝트 구조** | 파일이 여러 개일 때 **역할 설명** |

### 작성 예시 (목차)

```markdown
# [프로젝트 이름]

한 줄로 무엇을 하는 앱인지 설명

## 실행 방법

pip install streamlit pandas
streamlit run app.py

브라우저에서 http://localhost:8501 로 접속

## 기능
- (기획서와 같은 목록)

## 프로젝트 구조

app.py    ← Streamlit 앱 (실행하는 파일)

## 예외 처리
- 파일이 없을 때, 오류 날 때 어떻게 보이는지

## 범위 외
- 아직 구현하지 않은 기능
```

### 흔한 실수와 결과

| 실수 | 결과 |
|------|------|
| 실행 방법 없음 | README 품질 **심사 불가** 또는 낮은 점수 |
| 기획서와 다른 기능 설명 | **정합성** 감점 |
| `app.py`만 있고 구조 설명 없음 (다중 파일 프로젝트) | AI가 전체 구조를 **추측**해야 해서 불리 |

### 다른 사람에게 설명할 때

> 「README는 **심사위원이 우리 앱을 대신 실행해 보는 설명서**예요.  
> **복사해서 터미널에 붙여 넣으면 돌아가는 명령**까지 적어야 점수가 잘 나옵니다.」

## 4. 실행 코드 (`app.py`)

### 어디에 / 무엇을

- **채점 페이지 → 「실행 코드 (app.py)」** 칸
- 확장자: **`.py`만** (이름은 `app.py` 권장)

### 왜 `app.py` 하나만 올리나요?

채점 화면은 **파일 1개**만 받도록 되어 있습니다.  
그래서 **대표 진입 파일**(보통 `app.py`)을 올리고, 나머지 파일은 **README에 구조로 설명**하는 방식이 가장 정확합니다.

| 프로젝트 형태 | 권장 제출 방법 |
|---------------|----------------|
| `app.py` 한 파일에 전부 | 그 파일 업로드 ✅ |
| `app.py` + `utils.py` 등 여러 파일 | **`app.py` 업로드** + README에 **다른 파일 역할** 상세 기재 |

**왜 문제가 되나요?**  
`app.py`가 `from utils import …`처럼 **다른 파일을 부르는데** README에 설명이 없으면, AI는 `utils.py` 내용을 **볼 수 없어** 구현 여부를 **추측**만 합니다 → **의도 구현도 불리**.

### 흔한 실수와 결과

| 실수 | 결과 |
|------|------|
| 기획 기능 미구현 | 의도 구현도 **감점** |
| 기획에 없는 기능 다수 | 간결성·의도 구현도 **감점** |
| Python이 아닌 메모만 있음 | 코드 분야 **심사 불가** |
| 다른 과제 코드 | 의도 구현도 **급락** |

## 5. 파일 형식 — 왜 Word/PDF는 안 되나요?

| 형식 | 채점 가능? | 이유 |
|------|------------|------|
| `.md`, `.txt` | ✅ | **글자 그대로** 읽을 수 있음 |
| `.py` | ✅ | Python **소스 코드** 텍스트 |
| `.docx`, `.hwp` | ❌ | 워드 프로세서 **전용 형식**, 자동 추출 안 됨 |
| `.pdf` | ❌ | 레이아웃·이미지 혼합, 텍스트 추출 불안정 |
| `.zip` | ❌ | 압축 묶음, 내용 자동 분석 안 됨 |

**변환 방법 (Word 사용자)**

1. Word에서 **다른 이름으로 저장**
2. **일반 텍스트(.txt)** 또는 Markdown으로 저장
3. 한글이 깨지면 VS Code 등에서 **UTF-8**로 다시 저장

## 6. 제출 전 체크리스트

| # | 확인 | 안 지키면 |
|---|------|-----------|
| 1 | 세 파일이 **같은 프로젝트** | 의도·정합성 평가 불가 |
| 2 | README대로 **로컬에서 실행**됨 | README·구현 불일치 감점 |
| 3 | 기획 **성공 기준**이 코드에 있음 | 의도 구현도 감점 |
| 4 | 기획 **범위 외** 기능 없음 | 간결성 감점 |
| 5 | README에 **설치·실행·구조** 있음 | README 품질 감점 |
| 6 | 확장자 `.md`/`.txt`/`.py` | 업로드·심사 실패 |
| 7 | placeholder·빈 파일 아님 | 해당 분야 0점 |
| 8 | UTF-8로 한글 정상 | AI 오해·엉뚱한 평가 |

## 7. 4대 핵심 가치와 파일 연결

**채점 기준** 페이지의 4대 가치를 파일 준비와 이렇게 연결할 수 있습니다.

| 핵심 가치 | 파일에서 할 일 | 어기면 |
|-----------|----------------|--------|
| **정확성** | 기획 성공 기준 ↔ `app.py` 일치 | 의도 구현도 ↓ |
| **독창성** | 기획서에 현장 문제·차별 해결책 | 공공기관 적합성 ↓ |
| **완전성** | README만으로 설치·실행 가능 | README 품질 ↓ |
| **간결성** | 불필요 코드·기능 없음 | 간결성·의도 구현도 ↓ |

---

## 8. 팀 내에서 공유할 때 한 줄 요약

> 「**기획서 = 약속**, **app.py = 결과물**, **README = 사용 설명서**야.  
> 셋이 **같은 프로젝트** 이야기해야 하고, **글자 파일(.md/.txt/.py)** 로 올려야 해.  
> Word는 txt/md로 바꾸고, 코드는 **app.py** 올리되 여러 파일이면 README에 구조 적어.  
> **심사 시작은 한 번만** 눌러.」
